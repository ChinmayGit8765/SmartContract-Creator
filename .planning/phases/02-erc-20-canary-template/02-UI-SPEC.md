> Visual and interaction contract for the ERC-20 wizard flow. "UI" here means **terminal CLI UX** — prompt sequence, output channel routing, ANSI palette, three-part error blocks, and locked copy. Generated by gsd-ui-researcher, consumed by gsd-planner / gsd-executor / gsd-ui-checker.
>
> **NOT web/GUI.** Sections of the standard UI-SPEC template that assume a graphical surface (px spacing, color contrast ratios, breakpoints, touch targets, component registries) are repurposed for their CLI analogues or marked **N/A — terminal CLI**.

## Wizard Prompt Sequence (Locked Copy)

All copy is normative. The executor pastes these strings verbatim into source. `{}` placeholders are interpolated; nothing else changes.

### Pre-flight (newbie mode only, fires before prompt 1)

```text
output.explain("ERC-20 is the fungible-token standard on Ethereum and EVM chains. This wizard asks for the basics, then a few optional features.")
output.reference("EIP-20 spec", "https://eips.ethereum.org/EIPS/eip-20")
output.reference("OpenZeppelin ERC20 docs", "https://docs.openzeppelin.com/contracts/5.x/erc20")
```

Rendered (newbie mode, color on):
```text
ERC-20 is the fungible-token standard on Ethereum and EVM chains. This wizard asks for the basics, then a few optional features.
  see: EIP-20 spec (https://eips.ethereum.org/EIPS/eip-20)
  see: OpenZeppelin ERC20 docs (https://docs.openzeppelin.com/contracts/5.x/erc20)
```

In default mode: silent (none of the three lines render).
Under `--json`: silent (all three suppressed by `makeOutput` gate).

### Prompt 1 — Contract name

| Field | Value |
|-------|-------|
| `@clack` primitive | `text` |
| Message | `Contract name (Solidity identifier)` |
| Placeholder | `MyToken` |
| `defaultValue` | `MyToken` |
| Validator | `isSolidityIdentifier` (returns `"Must start with a letter or underscore, contain only letters/digits/underscores, max 64 chars."` on fail; `"Contract name is required."` on empty) |
| Newbie explain (before prompt) | `The on-chain contract name. Letters, digits, underscores; must start with a letter or underscore. Max 64 chars.` |

Validator regex: `/^[A-Za-z_][A-Za-z0-9_]{0,63}$/` (Solidity grammar, minus `$` for cross-language hygiene — see RESEARCH §Validators).

### Prompt 2 — Token symbol

| Field | Value |
|-------|-------|
| `@clack` primitive | `text` |
| Message | `Token symbol (1-11 ASCII letters/digits)` |
| Placeholder | `MTK` |
| `defaultValue` | `MTK` |
| Validator | `isAsciiSymbol` (returns `"Must be 1-11 ASCII letters/digits, no spaces or punctuation."` on fail; `"Token symbol is required."` on empty) |
| Newbie explain (before prompt) | `Wallets display this. 3-5 chars is conventional; the spec allows up to 11.` |

Validator regex: `/^[A-Za-z0-9]{1,11}$/`.

### Prompt 3 — Initial supply (premint)

| Field | Value |
|-------|-------|
| `@clack` primitive | `text` |
| Message | `Initial supply (human-readable, e.g. 1000000 or 1.5)` |
| Placeholder | `1000000` |
| `defaultValue` | `1000000` |
| Validator | `isNonNegativeDecimal` (returns `"Must be a non-negative decimal number, e.g. 1000000 or 1.5."` on fail; `"Initial supply is required (use 0 for no premint)."` on empty) |
| Newbie explain (before prompt) | `Minted to the deployer at deploy time. The wizard scales this by the token's decimals (default 18) — type the human-readable count, not the wei amount.` |

Validator regex: `/^(?:0|[1-9]\d*)(?:\.\d+)?$/`.

Special handling: `premint === "0"` is mapped to `undefined` before passing to `erc20.print()` to avoid emitting a `_mint(recipient, 0 * 10 ** decimals())` no-op line (RESEARCH Pitfall 5).

### Prompt 4 — Mintable

| Field | Value |
|-------|-------|
| `@clack` primitive | `confirm` |
| Message | `Enable Mintable? (an authorized account can mint more tokens after deploy)` |
| `initialValue` | `false` |
| Newbie explain (before prompt) | `Mintable means the supply is NOT fixed — an authorized account can mint more tokens later. Leave off for a hard-capped token.` |

### Prompt 5 — Burnable

| Field | Value |
|-------|-------|
| `@clack` primitive | `confirm` |
| Message | `Enable Burnable? (holders can burn their own tokens)` |
| `initialValue` | `false` |
| Newbie explain (before prompt) | `Burnable lets any holder destroy their own tokens (e.g., for deflationary models). It does NOT let an owner burn other people's tokens.` |

### Prompt 6 — Pausable

| Field | Value |
|-------|-------|
| `@clack` primitive | `confirm` |
| Message | `Enable Pausable? (an authorized account can freeze all transfers)` |
| `initialValue` | `false` |
| Newbie explain (before prompt) | `Pausable lets an authorized account freeze transfers in emergencies. Adds centralization risk — a single key can halt your token.` |

### Prompt 7 — Access control (conditional)

Fires **only if** `mintable || pausable`. Mirrors `erc20.isAccessControlRequired(opts)`. Per RESEARCH Assumption A6, only two options are surfaced (`ownable`, `roles`); the wizard SDK's third option `managed` is intentionally omitted as out-of-scope for v1.

| Field | Value |
|-------|-------|
| `@clack` primitive | `select` |
| Message | `Access control style:` |
| Options | `[{ value: "ownable", label: "Ownable — one address controls Mint/Pause" }, { value: "roles", label: "AccessControl — separate MINTER_ROLE / PAUSER_ROLE / DEFAULT_ADMIN_ROLE" }]` |
| Default | `ownable` (first option) |
| Newbie explain (before prompt) | `Ownable: one address controls Mint/Pause. Simpler but a single key controls the contract. AccessControl: separate roles for MINTER_ROLE / PAUSER_ROLE / DEFAULT_ADMIN_ROLE. More flexible, more setup. Use AccessControl if you plan to use a multisig or split duties.` |

If the prompt does not fire (`!mintable && !pausable`), the returned `Erc20Opts.access` is `false`.

### Post-prompt — Centralization warning (UI-02)

Runs after prompt 7 resolves (or is skipped), before `runWizard()` returns.

```ts
if (opts.mintable && opts.access === "ownable") {
  output.warn(
    "Mintable + Ownable: a single key can mint unlimited tokens. " +
    "Consider AccessControl (multi-role) or transferring ownership to a multisig before deploy."
  );
}
```

Rendered (color on):
```text
warn: Mintable + Ownable: a single key can mint unlimited tokens. Consider AccessControl (multi-role) or transferring ownership to a multisig before deploy.
```

Fires in **all** modes (default, newbie, `--json`) — `output.warn` is the always-on critical-warning channel per Phase 1 contract.

## Output Channel Routing Map

For each event in the Phase 2 happy path, this table locks which `output.*` channel is called and which modes render the line.

| Event | Channel | Default | `--newbie` | `--json` | Stream |
|-------|---------|---------|------------|----------|--------|
| Newbie preamble explain | `output.explain` | silent | render (dim) | silent | stdout |
| Newbie preamble references (×2) | `output.reference` | silent | render (cyan `see:` + dim url) | silent | stdout |
| Per-prompt explain (×7 conditional) | `output.explain` | silent | render (dim) | silent | stdout |
| Prompt rendering itself | (clack) | render | render | render (BUT see UI-10 — refused) | tty |
| Centralization warning (UI-02) | `output.warn` | render (yellow `warn:`) | render | render | stderr |
| Validator failure (inline re-prompt) | (clack `validate`) | render inline | render inline | N/A (UI-10) | tty |
| Wizard cancel (Ctrl+C) | thrown CliError → `renderError` → `output.error` | render three-part block | render three-part block | render three-part block | stderr |
| Overwrite prompt (`confirmOverwrite`) | (clack) | render | render | N/A (UI-10) | tty |
| Overwrite refused | thrown CliError → `renderError` → `output.error` | render | render | render | stderr |
| File written | `output.result` | render `Wrote {path}` | render `Wrote {path}` | render `Wrote {path}` (callers wrap JSON before passing — see Phase 1 contract) | stdout |
| Post-write nextStep (×2) | `output.nextStep` | silent | render (green `next:`) | silent | stdout |

Channel semantics (already locked Phase 1, restated for executor convenience):

- `result` — always on; stdout; byte-passthrough (no prefix, no color).
- `warn` — always on; stderr; `warn:` prefix in yellow.
- `error` — always on; stderr; caller-formatted (we use `renderError(CliError, colors)`).
- `explain` — newbie AND not-json; stdout; dim.
- `reference(label, url)` — newbie AND not-json; stdout; `  see: <label> (<url>)`, `see:` in cyan, url parenthetical in dim.
- `nextStep` — newbie AND not-json; stdout; `  next: <text>`, `next:` in green.

## Post-Generation Footer (Locked Copy)

Runs after `await writeFile(outPath, source, "utf8")` resolves successfully.

```ts
output.result(`Wrote ${outPath}`);
output.nextStep("Run 'smartc list-templates' to see other templates.");
output.nextStep("Phase 3 will add automatic compile-verify before write — for now the .sol references @openzeppelin/contracts which you'll need installed to compile.");
```

Rendered in newbie mode:
```text
Wrote /path/to/MyToken.sol
  next: Run 'smartc list-templates' to see other templates.
  next: Phase 3 will add automatic compile-verify before write — for now the .sol references @openzeppelin/contracts which you'll need installed to compile.
```

Rendered in default mode:
```text
Wrote /path/to/MyToken.sol
```

**Honesty framing rules (UI-05):**

- DO say "Wrote {path}". DO NOT say "Created", "Generated", or "Built" — they imply more than a `fs.writeFile`.
- DO NOT say "Ready to deploy". DO NOT say "Compile-verified". The Phase 3 gate is not wired yet.
- DO point at Phase 3 as a **future** capability ("will add"), not a current one.
- DO NOT mention Phase 5 DEPLOY.md in Phase 2's nextStep — that's an even further-future capability and we keep the footer to two lines max for clarity.

## Three-Part Error Block Copy (Phase 2 New Codes)

All three new/used codes render through `renderError(err, color)` (locked Phase 1). The block shape is invariant:

```text
Error: <what>  (code: <code>)
Why:   <why>
Fix:   <fix>
```

Colors (locked Phase 1, restated): `Error:` red, `Why:` yellow, `Fix:` cyan, code parenthetical dim.

### E_WIZARD_CANCEL (new — Phase 2 ships)

Thrown by `cancelGuard(answer, promptName)` inside `runWizard()` when `@clack/prompts.isCancel(answer)` returns true.

| Field | Value |
|-------|-------|
| `code` | `E_WIZARD_CANCEL` (constant: `ERR_WIZARD_CANCEL`) |
| `exitCode` | `130` (Unix SIGINT convention) |
| `what` | `Wizard cancelled at: {promptName}.` |
| `why` | `You pressed Ctrl+C or otherwise dismissed the prompt.` |
| `fix` | `Re-run 'smartc create --template erc20' to start over.` |

`{promptName}` is one of: `"contract name"`, `"token symbol"`, `"initial supply"`, `"mintable"`, `"burnable"`, `"pausable"`, `"access control"`.

Rendered:
```text
Error: Wizard cancelled at: token symbol.  (code: E_WIZARD_CANCEL)
Why:   You pressed Ctrl+C or otherwise dismissed the prompt.
Fix:   Re-run 'smartc create --template erc20' to start over.
```

### E_INVALID_INPUT (new — Phase 2 ships the constant; not thrown on happy path; reserved for forward use — UI-15)

Add to `src/lib/errors.ts`:

```ts
export const ERR_INVALID_INPUT = "E_INVALID_INPUT" as const;
```

Phase 2 does NOT construct this CliError in production code — `@clack/prompts` validators handle inline re-prompting. The code is reserved so Phase 3+ (compile diagnostics surfaced as input failures) and the future deferred flag-driven non-interactive mode can throw it without renaming. **Locked template for forward callers:**

| Field | Value template |
|-------|----------------|
| `code` | `E_INVALID_INPUT` |
| `exitCode` | `1` (default) |
| `what` | `Invalid {field}: {value}.` |
| `why` | `{specific reason — e.g., "Must be a Solidity identifier (letters, digits, underscores; must start with a letter or underscore)."}` |
| `fix` | `Re-run interactively without --{field-flag}, or supply a valid value.` |

### E_FILE_EXISTS (already shipped Phase 1 — Phase 2 first real consumer)

Thrown by `confirmOverwrite(path, { force })` when the user answers no or cancels the overwrite prompt. Already locked in `src/lib/prompt.ts`; Phase 2 changes nothing.

| Field | Value |
|-------|-------|
| `code` | `E_FILE_EXISTS` |
| `exitCode` | `1` |
| `what` | `Refused to overwrite {path}.` |
| `why` | `The output path already exists and you chose not to overwrite it.` |
| `fix` | `Re-run with a different --out path, or pass --force to overwrite without prompting.` |

### E_USAGE (already shipped Phase 1 — Phase 2 reuses for `--json` refusal per UI-10)

When the user passes `--json` to `create`, the dispatcher throws this BEFORE invoking the registry or the wizard:

| Field | Value |
|-------|-------|
| `code` | `E_USAGE` |
| `exitCode` | `2` (matches Phase 1 commander-usage-error convention) |
| `what` | `'smartc create' cannot run in --json mode.` |
| `why` | `The wizard requires an interactive TTY, which is incompatible with machine-readable output.` |
| `fix` | `Re-run without --json. Flag-driven non-interactive generation is planned for a future release; track it in .planning/STATE.md.` |

## ANSI Palette + Degradation

Phase 1's `makeColor(noColorFlag)` exposes six functions. Phase 2 uses five of them across `output.*` and `renderError`. No new color helpers added.

| Color | Role in Phase 2 | Used by | Degrades to |
|-------|-----------------|---------|-------------|
| `red` | Critical: error label | `Error:` in three-part block | Plain `Error:` text |
| `yellow` | Caution: warning label | `warn:` prefix; `Why:` in three-part block | Plain `warn:` / `Why:` text |
| `cyan` | Pointer: hyperlink-ish | `see:` in references; `Fix:` in three-part block | Plain `see:` / `Fix:` text |
| `green` | Affirmation: forward-motion | `next:` in nextStep | Plain `next:` text |
| `dim` | De-emphasis: hint | `explain` body, url parentheticals, code parenthetical | Plain text |
| `bold` | (unused in Phase 2) | (`list-templates` headers — inherited unchanged) | Plain text |

### Degradation contract (locked Phase 1, restated)

`makeColor` enables ANSI iff `!noColorFlag && pc.isColorSupported`. `pc.isColorSupported` already honors:

- `NO_COLOR` env (https://no-color.org/)
- `FORCE_COLOR` env
- `process.stdout.isTTY`
- CI detection (CI=1 / GitHub Actions / etc., with picocolors' own heuristics)

When disabled, every color function returns its input string unchanged (identity). Phase 2 does NOT add any other escape sequences (no cursor moves, no clearing, no spinners outside what @clack already manages internally). The output is byte-clean under `--no-color` and `NO_COLOR=1`.

**Verification (already covered by Phase 1's e2e):** `tests/cli.spec.ts` spawns the binary with `NO_COLOR=1` and asserts no `[` ESC sequences appear in stderr. Phase 2 adds no new color-using code paths that need separate verification.

## `--json` Mode Contract for `create` (UI-10)

Phase 2 does NOT support `create --json`. The wizard requires a TTY and Phase 2's deferred-ideas list excludes flag-driven non-interactive generation. The dispatcher refuses early:

```ts
// In src/commands/create.ts .action(), before resolving the template:
if (globalOpts.json) {
  throw new CliError({
    code: ERR_USAGE,
    what: "'smartc create' cannot run in --json mode.",
    why: "The wizard requires an interactive TTY, which is incompatible with machine-readable output.",
    fix: "Re-run without --json. Flag-driven non-interactive generation is planned for a future release; track it in .planning/STATE.md.",
  });
}
```

Exit code: 2 (per E_USAGE policy). The error block renders to stderr as plain text (red `Error:` + yellow `Why:` + cyan `Fix:` if color is on); under `--no-color`, plain text. **The error block is NOT serialized to JSON** — we use the same human-readable block as the rest of the CLI for consistency. JSON-mode failures are a corner case where the user is misusing the tool; making them parse our error JSON is over-engineering for a refused-up-front case.

**Coordination seam for future flag-driven mode (Deferred):** when the future `smartc create --template erc20 --name X --symbol Y --supply N --mintable --access roles` lands, the dispatcher will skip the TTY check, run a non-interactive opts builder (no @clack calls), throw `E_INVALID_INPUT` for invalid flag values (per UI-15), and emit `{"path": "/path/to/MyToken.sol"}` as `output.result(JSON.stringify(...))`. The newbie channels stay silent under `--json` (Phase 1 contract). UI-15's locked `E_INVALID_INPUT` template is the contract that mode will honor.

## Spacing / Indent Hierarchy (CLI Analogue)

Terminals don't have pixel spacing, but the project uses consistent indent and blank-line conventions for readability. Locked from Phase 1's existing output and extended only where Phase 2 introduces new lines.

| Element | Indent | Trailing newline | Notes |
|---------|--------|------------------|-------|
| `output.result` line | 0 spaces | yes (factory appends if missing) | "Wrote {path}" |
| `output.warn` line | 0 spaces | yes | `warn: <text>` |
| `output.explain` line | 0 spaces | yes | dim body |
| `output.reference` line | 2 spaces | yes | `  see: ...` |
| `output.nextStep` line | 2 spaces | yes | `  next: ...` |
| Three-part error block | 0 spaces; `Why:` and `Fix:` aligned via two trailing spaces in label | yes (one line per part) | `Error: ... \n Why:   ... \n Fix:   ...` |
| Between wizard prompts | (clack manages) | — | One visual line gap between prompts; do not inject our own blank lines |
| Between newbie preamble and prompt 1 | none (clack opens prompt directly after our writes) | — | If readability suffers, planner may add one `output.result("")` blank-line emit — but default is no extra blank lines |
| Between centralization warning and next prompt | none | — | The warning fires after prompt 7 resolves; the next event is `runWizard()` return — no further wizard output |
| Between post-write `result` and `nextStep` | one line break (each line ends `\n`) | yes | No blank line between; the visual indent of `  next:` provides the hierarchy |

**Rule:** Phase 2 does not emit blank-line separators between channels. The two-space indent on `reference` / `nextStep` IS the visual hierarchy.
